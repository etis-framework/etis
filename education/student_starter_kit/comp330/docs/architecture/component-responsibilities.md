# Component Responsibilities

## Purpose

This document defines ownership and responsibilities for major system components.

Responsibilities should be clear.

Responsibilities should be understandable.

Responsibilities should minimize unnecessary overlap.

Components should have focused purposes.

## Dependency Notes

Document important dependencies.

Examples:

- User interface depends on application services.
- Application services depend on data storage.
- Reporting depends on request data.

---

## Boundary Notes

Document any important boundaries.

Examples:

- Validation occurs before persistence.
- External services are isolated.
- User permissions are enforced before actions execute.

---

## Review Questions

- Are responsibilities clear?
- Is ownership visible?
- Is unnecessary overlap emerging?
- Are dependencies understandable?

---

## Engineering Standard

Strong systems have clear responsibilities.
