# Acceptance Criteria

## Purpose

This document defines the evidence needed to determine whether requirements have been satisfied.

Acceptance criteria connect requirements to review, testing, release readiness, and final engineering defense.

They should be specific enough that another reviewer can determine whether the system meets the requirement.

## Acceptance Criteria Format

Use this format for each acceptance criterion.

```text
Acceptance Criterion ID:
Related Requirement:
Condition:
Expected Behavior:
Evidence Required:
Verification Method:
Status:
Related Test Evidence:
```

## Verification Methods

Possible verification methods include:

* inspection
* demonstration
* unit test
* integration test
* user acceptance test
* review
* analysis
* operational simulation

## Review Questions

Before considering acceptance criteria ready, answer:

* Can the criterion be verified?
* Is the expected behavior clear?
* Is the required evidence identified?
* Does the criterion avoid vague terms such as “easy,” “fast,” or “user-friendly” unless measured?
* Is the criterion connected to a requirement?
* Is the criterion connected to test evidence?

## Engineering Standard

Acceptance criteria should make engineering claims reviewable.

A team should not claim a requirement is satisfied unless acceptance evidence exists.
