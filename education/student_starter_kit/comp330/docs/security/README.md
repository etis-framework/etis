# Security Package

## Purpose

This directory helps teams make security assumptions visible throughout the project.

Security is not a separate engineering activity performed at the end.

Security should be considered continuously as systems evolve.

Keep security lightweight and practical.

Do not over-engineer it.

---

## Security Artifacts

| Artifact | Purpose |
|----------|---------|
| `data-handling-notes.md` | Documents how data is handled and protected |
| `permission-boundaries.md` | Documents access boundaries and responsibilities |
| `security-governance-checklist.md` | Provides a lightweight security review checklist |

---

## Relationship To ETIS Philosophy

Security is the practice of making risks visible before they become failures.

Students should continuously ask:

- What data are we handling?
- Who should have access?
- What assumptions are we making?
- What risks remain?

Security should become routine engineering behavior.

---

## Security Progression

This directory intentionally supports the following progression.

```text
Data Protection

↓

Access Boundaries

↓

Security Governance
```

Strong engineering teams make security assumptions visible.

---

## Review Questions

Teams should periodically ask:

- Are we handling data responsibly?
- Are access boundaries clear?
- Are security assumptions documented?
- Are obvious risks visible?

---

## Engineering Standard

Strong engineering teams make security visible before release.
