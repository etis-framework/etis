# ETIS Assignment Sequence Guidance

The **ETIS Assignment Sequence Guidance** directory provides reusable models for organizing educational experiences around progressive engineering accountability.

This directory does not provide collections of assignments.

It provides systems for sequencing engineering maturity through cumulative phase gates.

ETIS intentionally moves beyond assignment-centric education.

The objective is not to help students complete work.

The objective is to help students progressively assume responsibility for engineering outcomes.

---

## Purpose

The purpose of this directory is to help instructors design educational experiences that resemble professional engineering environments.

This directory helps instructors answer questions such as:

- How should engineering accountability increase over time?
- How should assignments build upon one another?
- How should evidence accumulate?
- How should reviews evolve?
- How should students progressively defend engineering decisions?
- How should educational work resemble professional engineering work?
- How should phase gates be assembled?
- How should assignments adapt to different educational environments?

The objective is to sequence professional accountability rather than distribute assignments throughout a course.

---

## Educational Philosophy

Traditional courses often organize learning around independent assignments.

A common progression looks like this:

```text
Assignment 1

↓

Assignment 2

↓

Assignment 3

↓

Final Project
```

Each assignment often exists independently.

ETIS intentionally organizes learning differently.

```text
Repository Foundation

↓

Intent

↓

Architecture

↓

Construction

↓

Release Readiness

↓

Operations

↓

Engineering Defense
```

Each stage inherits from previous work.

Nothing is truly independent.

Students should repeatedly improve engineering artifacts rather than repeatedly create new artifacts.

---

## Guiding Principle

ETIS assignment sequencing should answer one question:

> What new engineering responsibility should students now be able to own?

Students should progressively learn how to:

- define intent,
- engineer context,
- bound authority,
- verify behavior,
- operate reality,
- explain decisions,
- and own outcomes.

Assignments should support that progression.

Assignments should not become the progression itself.

## What Does Not Belong Here

This directory should remain reusable and implementation agnostic.

The following artifacts should not be placed here:

- semester-specific assignments,
- year-specific assignments,
- instructor-specific assignments,
- local grading spreadsheets,
- institutional calendars,
- LMS exports,
- COMP330 semester schedules,
- administrative logistics.

Those artifacts belong inside adoption examples.

For example:

```text
education/
└── adoption_examples/
    └── loyola_comp330/
```

Examples include:

- quarter systems,

- project complexity,
- instructional modalities,
- institutional requirements.

These constraints may alter pacing.

They should not alter ETIS doctrine.

## Stewardship Rules

Future maintainers should preserve these rules.

- Do not create assignment libraries.
- Do not create isolated assignments.
- Do not disconnect assignments from previous work.
- Do not duplicate COMP330 implementations.
- Do not prioritize deliverables over accountability.
- Do not disconnect assignments from ETIS doctrine.

This directory should remain cumulative, reusable, and maintainable.

---

## Guiding Standard

Every file in this directory should answer at least one of these questions:

- What engineering responsibility is being taught?
- What evidence should students create?
- What should students review?
- What should students defend?
- What should students improve?
- What should students own?

If a file cannot answer one of those questions, it probably does not belong here.
