# ETIS Syllabus Guidance

The **ETIS Syllabus Guidance** directory provides reusable syllabus components that instructors can assemble into ETIS-based courses.

This directory does **not** provide a single ETIS syllabus.

ETIS is intentionally adaptable to different institutions, audiences, calendars, instructional models, and educational goals.

Instead of prescribing one syllabus, ETIS provides reusable building blocks that instructors can assemble into their own implementations while preserving alignment with ETIS doctrine.

---

## Purpose

The purpose of this directory is to help instructors efficiently build syllabi that reflect the educational philosophy of *Engineering Trustworthy Intelligent Systems (ETIS)*.

This directory supports instructors by providing reusable guidance for:

* course descriptions,
* learning outcomes,
* AI-use policies,
* repository expectations,
* professional expectations,
* academic integrity language,
* and syllabus assembly.

The objective is to reduce course preparation effort while maintaining consistency across ETIS implementations.

---

## Guiding Principle

ETIS syllabi should describe the formation of trustworthy engineers rather than simply list topics, assignments, and grading policies.

An ETIS syllabus should communicate that software engineering is a professional discipline grounded in:

* engineering intent,
* evidence creation,
* architecture reasoning,
* AI accountability,
* review participation,
* operational thinking,
* governance,
* and long-term stewardship.

Students should understand from the first day of class that ETIS extends beyond software construction.

## Syllabus Assembly Philosophy

ETIS syllabi should be assembled intentionally.

A typical process may follow this sequence:

```text
1. Select course type

2. Select professional transformation goals

3. Select learning outcomes

4. Select ETIS chapters

5. Select AI policies

6. Select repository expectations

7. Select professional expectations

8. Select assessment models

9. Assemble the syllabus
```

This process allows instructors to adapt ETIS without recreating the educational ecosystem from scratch.

## What Does Not Belong Here

This directory should remain reusable and institution-agnostic.

The following artifacts should not be placed here:

* semester-specific syllabi,
* instructor-specific syllabi,
* Fall or Spring syllabi,
* institution-specific schedules,
* implementation-specific grading schemes,
* COMP330-specific documents,
* or local administrative policies.

Those artifacts belong inside course implementations.

For example:

```text
education/
└── adoption_examples/
    └── loyola_comp330/
```

## Institutional Adaptation

Institutions may adapt ETIS syllabi to fit local constraints.

Examples include:

* semester length,
* quarter systems,
* class size,
* institutional policies,
* departmental requirements,
* technology choices,
* and assessment models.

These adaptations should preserve ETIS doctrine while allowing implementation flexibility.

---

## Stewardship Rules

Future maintainers should preserve these rules.

* Do not create complete syllabi in this directory.
* Do not duplicate implementation artifacts.
* Do not duplicate COMP330 materials.
* Do not make assets semester-specific.
* Do not create unnecessary administrative language.
* Do not disconnect syllabi from ETIS doctrine.

This directory should remain small, reusable, and maintainable.

---

## Guiding Standard

Every file in this directory should answer at least one of these questions:

* How should ETIS be described to students?
* What engineering responsibilities should students understand?
* What professional expectations should students know?
* How should AI-assisted work be governed?
* What evidence should students create?
* How should instructors assemble a syllabus responsibly?

If a file cannot answer one of those questions, it probably does not belong here.

---

## Core Commitment

The purpose of ETIS syllabus guidance is not to standardize every ETIS course.

The purpose is to help instructors communicate a consistent educational philosophy while preserving flexibility for local implementation.

Students should leave an ETIS course understanding that software engineering is not simply the act of producing software.

It is the disciplined practice of creating systems that can be understood, reviewed, governed, operated, improved, and trusted over time.
