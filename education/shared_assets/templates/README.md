# Templates

## Purpose

The `templates` directory contains educational assets that help learners create structured engineering artifacts throughout the software engineering lifecycle.

The purpose of these assets is to provide engineering scaffolding while preserving student ownership of the work itself.

Templates guide students toward creating professional engineering evidence rather than simply producing deliverables.

Software engineering is a discipline built upon artifacts that support communication, decision making, governance, collaboration, review, operation, and long-term stewardship.

Templates make these expectations explicit.

---

## Mission

The mission of `templates` is to teach engineers how to create durable, understandable, reviewable, and maintainable engineering artifacts.

These assets help learners understand that software engineering extends far beyond writing code.

Professional engineers routinely create artifacts that help others understand systems, evaluate decisions, manage risks, and operate software over time.

The objective is not to fill out forms.

The objective is to teach engineers how to create meaningful engineering evidence.

---

## Educational Philosophy

Engineering work should be understandable by people other than the original author.

Organizations depend upon artifacts to preserve knowledge and coordinate work across teams.

Professional engineers routinely create artifacts that answer questions such as:

* What are we building?
* Why are we building it?
* What assumptions were made?
* What risks exist?
* How was the system designed?
* How was quality evaluated?
* How should the system be operated?
* What decisions were made?

Templates intentionally teach students how to answer these questions.

## Foundational ETIS Principles

The following ETIS principles strongly influence template design.

### AI proposes; engineers verify.

Artifacts should document human decisions and verification activities.

### Governance is architecture.

Engineering artifacts help operationalize governance.

### Context is control.

Artifacts preserve context over time.

### Everything important leaves evidence.

Templates exist to help engineers create evidence.

### The model is not the system.

Artifacts should represent broader system thinking.

### A demo is not operational proof.

Engineering artifacts should support long-term sustainability.

* Planning artifacts
* Design artifacts
* Review artifacts
* Testing artifacts
* Operational artifacts
* Governance artifacts
* Documentation frameworks

Templates should guide engineering thinking without prescribing engineering decisions.

---

## What Does Not Belong Here

The following do not belong in this directory:

* Behavioral guidance
* Workflow instructions
* Grading rubrics
* Tool installation instructions
* Repository setup guides
* Assignment specifications

Those assets belong elsewhere within the ecosystem.

Templates should remain focused on engineering evidence creation.

## Engineering Evidence Philosophy

Templates exist because engineering work should survive beyond the original engineer.

Strong engineering artifacts help future engineers answer questions without requiring direct access to the original team.

Good engineering artifacts should support:

* Communication
* Collaboration
* Review
* Governance
* Operations
* Continuous improvement

The artifact itself becomes part of the engineering system.

---

## Governance Rules

The following rules apply to all assets within this directory.

### Prioritize engineering evidence.

Templates should help engineers preserve meaningful information.

### Encourage explainability.

Artifacts should be understandable by others.

### Support reviewability.

Artifacts should enable evaluation and discussion.

### Preserve context.

Artifacts should communicate assumptions and decisions.

### Remain implementation-neutral.

Templates should be reusable across implementations.

### Promote long-term stewardship.

Artifacts should support future engineers.
