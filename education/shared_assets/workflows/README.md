# Workflows

## Purpose

The `workflows` directory contains educational assets that teach repository-centered engineering processes and the movement of engineering work throughout the software engineering lifecycle.

The purpose of these assets is to help learners understand that engineering work is not a collection of isolated activities.

Engineering work flows through a system of interconnected decisions, artifacts, reviews, implementations, and operational activities.

Workflows make these relationships visible.

These assets teach students how engineering work progresses from initial intent to long-term operation.

---

## Mission

The mission of `workflows` is to teach engineers how to organize, coordinate, execute, review, and sustain engineering work over time.

These assets help learners understand that successful engineering organizations rely upon repeatable processes rather than ad hoc execution.

Professional engineers work within systems of accountability, traceability, and collaboration.

The objective is not to teach students how to follow procedures.

The objective is to teach students how engineering systems operate.

---

## Educational Philosophy

Software engineering is a flow problem as much as it is a technical problem.

Engineering work continuously moves through a series of activities.

Intent becomes requirements.

Requirements become designs.

Designs become implementations.

Implementations become reviews.

Reviews become releases.

Releases become operations.

Operations become learning opportunities.

Students should intentionally learn these relationships.

Workflows make this movement visible.

---

## Foundational ETIS Principles

The following ETIS principles strongly influence workflow design.

### AI proposes; engineers verify.

Workflow checkpoints should preserve human oversight.

### Governance is architecture.

Processes operationalize governance.

### Context is control.

Workflows preserve and transfer context.

### Everything important leaves evidence.

Workflow activities should generate engineering evidence.

### The model is not the system.

Workflows should account for broader system realities.

### A demo is not operational proof.

Workflows should extend beyond implementation and demonstrations.

## What Does Not Belong Here

The following do not belong in this directory:

* Behavioral guidance
* Fill-in templates
* Grading rubrics
* Assignment specifications
* Tool installation instructions
* Vendor marketing materials

Those assets belong elsewhere within the ecosystem.

Workflows should remain focused on engineering execution systems.

## Repository-Centered Engineering Philosophy

Repository-centered engineering is a foundational concept within ETIS.

The repository is not a storage location.

The repository is the operational center of engineering activity.

The repository becomes the location where engineers:

* Define intent
* Create evidence
* Coordinate work
* Preserve context
* Conduct reviews
* Track progress
* Prepare releases
* Support operations

The workflows within this directory teach students how to use repositories as engineering systems rather than code containers.
